# PROJECT_BRAIN

## 1. Project Overview

- **Что это:** Одностраничное web-приложение для создания задач в Bitrix24 через REST API (входящий вебхук)
- **Задача:** Упростить и стандартизировать постановку задач в команде — выбрать разработчика, постановщика, проект, дедлайн, заполнить структурированное описание, прикрепить файлы — всё в одном клике
- **Стек технологий:**
  - HTML5 (один файл `index.html`, 788 строк)
  - CSS3 (Custom Properties, Grid, Flexbox, backdrop-filter, keyframes)
  - Vanilla JS (ES6+, async/await, Fetch API, FileReader API, XHR)
  - Google Fonts: JetBrains Mono + Unbounded
  - Inline SVG иконки (без библиотек)
  - LocalStorage для конфигурации
  - Bitrix24 REST API (входящий вебхук)
- **Репозиторий:** `https://github.com/Antisakrum2004/bitrix-form`
- **Текущая версия:** v6 (по бейджу в topbar)

## 2. Architecture

### Общая схема

```
┌──────────────────────────────────────────────┐
│  Браузер (index.html)                        │
│  ┌──────────┐  ┌──────────┐  ┌───────────┐  │
│  │  UI Form │→ │  JS App  │→ │ Bitrix24  │  │
│  │ (CSS/    │  │ (Vanilla │  │ REST API  │  │
│  │  HTML)   │  │  async)  │  │ (вебхук)  │  │
│  └──────────┘  └──────────┘  └───────────┘  │
│                     ↕                        │
│              LocalStorage                     │
│         (вебхук, последняя задача)           │
└──────────────────────────────────────────────┘
```

- **Backend:** Отсутствует. Прямые вызовы Bitrix24 REST API из браузера
- **Frontend:** Единый HTML-файл со встроенными CSS и JS
- **Интеграция:** Bitrix24 REST API через входящий вебхук (URL вида `https://ПОРТАЛ.bitrix24.ru/rest/1/ТОКЕН/`)

### Ключевые зависимости

- `Google Fonts CDN` — шрифты (JetBrains Mono, Unbounded)
- `Bitrix24 REST API` — все данные и операции
- Внешних npm-пакетов нет

### Как идут данные

1. Пользователь заполняет форму → JS собирает payload
2. `POST {webhook}/tasks.task.add` → создание задачи, получение `task.id`
3. `POST {webhook}/task.observers.add` (до 4 вариантов эндпоинта) → добавление наблюдателя
4. `GET {webhook}/disk.folder.uploadfile?id=3&uploadFile=...` → получение upload URL
5. Загрузка файла на upload URL (3 метода с фоллбэком)
6. `POST {webhook}/tasks.task.update` с `UF_TASK_WEBDAV_FILES: ['n'+diskId]` → привязка файла к задаче
7. Сброс формы, показ toast

- **Путь:** `.card` Card 5 (строки 334–339)

### Модуль: API-интеграция
- **Путь:** `index.html` → `<script>` функции `hS()`, `addObs()`, `uFL()`
- **Ответственность:**
  - Создание задачи через `tasks.task.add`

  - Загрузка файлов: 3-шаговый процесс (получить URL → загрузить → привязать)
  - 3 метода загрузки файлов с автофоллбэком: raw binary, XHR+FormData, fetch+FormData
- **Связи:** Bitrix24 REST API, LocalStorage

### Модуль: Toast-уведомления
- **Путь:** `index.html` → `.toast` + `showT()`
- **Ответственность:** Success/error уведомления, автоскрытие через 6 секунд
- **Связи:** Вызывается из всех модулей

## 5. Known Issues & Problems

- **CORS-ограничения:** Прямые `fetch()` к Bitrix24 API из `file://` не работают. Требуется запуск через локальный сервер (Live Server, `python -m http.server`). В коде есть подсказка при TypeError
- **Загрузка файлов — HTTP 400:** Историческая проблема (коммиты от 6 апреля). Multipart-загрузка через `disk.file.upload` возвращала HTTP 400 для `.txt`, `.xlsx`, `.epf`. Решено через 3 метода с автофоллбэком, но не гарантирует 100% успеха
- **Наблюдатель — нет единого рабочего эндпоинта:** Bitrix24 API не имеет стабильного метода для добавления наблюдателя через вебхук. Перебираются 4 варианта (`task.observers.add`, `tasks.task.observers.add`, `tasks.task.update` с `AUDITORS`). Если ни один не сработал — показывается предупреждение
- **Жёстко закодированные ID:** Массивы `DEVS`, `MGRS`, `OBSS`, `PROJS` содержат конкретные Bitrix24 user/group ID. При изменении состава команды/проектов — правка кода
- **Один файл:** Весь проект в одном `index.html` (788 строк). Сложно поддерживать и тестировать
- **Нет обработки ошибок сети:** Если вебхук недоступен — пользователь видит только toast «CORS» или «Ошибка сети»
- **Теги мёртвый код:** Переменная `tags`, функции `addTag()`, `removeTag()`, `rT()` существуют, но UI для тегов удалён. В payload передаётся `TAGS:tags` (всегда пустой массив)

## 7. Decisions & Reasoning

- **Vanilla JS без фреймворков:** Форма — единственная страница, состояние минимально. Ради одной формы нет смысла тянуть React/Vue. Нулевой бандл, мгновенная загрузка, полный контроль
- **Один HTML-файл:** Удобно деплоить — загрузил один файл на GitHub Pages / любой хостинг. Нет сборки, нет зависимостей
- **Вебхук через входящий URL:** Простейший способ авторизации в Bitrix24 без OAuth. Пользователь один раз вставляет URL — всё работает
- **LocalStorage для вебхука:** Не нужен сервер, нет проблем с безопасностью для внутреннего инструмента
- **Жёстко закодированные ID людей и проектов:** Команда и проекты стабильны. Нет смысла тянуть справочники из API при каждой загрузке — это замедлит форму
- **3 метода загрузки файлов с фоллбэком:** Bitrix24 API нестабилен при загрузке файлов через вебхук. Разные методы работают для разных типов файлов. Фоллбэк — единственный надёжный подход
- **Наблюдатель — отдельный запрос:** `AUDITORS` не работает в `tasks.task.add`, поэтому наблюдатель добавляется после создания задачи
- **Структурированное описание (секции):** Задачи в команде были неструктурированными. Форма заставляет заполнить цель, что сделать, критерии приёмки — повышает качество постановки
- **Убрать теги (v5):** Теги дублировали функцию проектов. Упрощение UI

## 8. Limitations / Tech Debt

- **Мёртвый код:** Переменная `tags`, функции `addTag()`, `removeTag()`, `rT()` — UI удалён, код остался. В payload: `TAGS:tags` (пустой массив)
- **Один монолитный файл:** 788 строк HTML+CSS+JS в одном файле. Нет модульности, нет тестов, сложно рефакторить
- **Минифицированные имена функций:** `ag()`, `ufi()`, `up()`, `sC()`, `sP()`, `rL()`, `hS()`, `hF()`, `rF()`, `bD()`, `oS()`, `cS()`, `mWS()`, `sWH()`, `uRB()`, `fSz()`, `tF()`, `esc()`, `showT()`. Нечитаемо без документации
- **Нет unit-тестов:** Нет никакого тестирования, ручная проверка
- **Нет сборки/транспиляции:** Нет Babel, нет бандлера. ES6+ может не работать в старых браузерах
- **Нет CI/CD:** Коммиты напрямую в main, нет пайплайна
- **CORS-зависимость:** Работает только через HTTP-сервер, не из `file://`
- **Нет TypeScript:** Нет типизации, легко допустить ошибку в структуре payload
- **Файлы на диск с folder_id=3:** Жёстко закодированный ID папки (`disk.folder.uploadfile?id=3`). При изменении структуры диска в Bitrix24 — сломается
- **Нет обработки параллелизма:** Если кликнуть «Создать» дважды быстро — создастся 2 задачи
- **Наблюдатель не гарантирован:** Перебор 4 вариантов — workaround, не решение

## 10. Quick Context for AI

### Как писать код в этом проекте
- Всё в одном файле `index.html`: CSS в `<style>`, JS в `<script>`, HTML в `<body>`
- Нет сборки, нет npm, нет бандлера. Прямой редактируемый HTML
- Проверяй изменения открытием файла через Live Server / `python -m http.server` (CORS!)
- Не добавляй внешние зависимости без крайней необходимости

### Принципы
- Минимализм: минимум зависимостей, нулевой бандл
- Практичность над красотой: форма должна работать надёжно, а не выглядеть идеально
- Фоллбэки: если что-то может не работать — предусмотрены варианты (как с загрузкой файлов)
- Bitrix24 API нестабилен: всегда перебирай несколько вариантов эндпоинта

### Стиль кода
- CSS: Custom Properties (`--accent`, `--bg`, и т.д.), краткие имена классов, glass-морфизм
- JS: ES6+, async/await, краткие/минифицированные имена функций (исторически сложилось), однобуквенные переменные в коротких функциях
- HTML: Semantic5 где возможно, но в основном div-ная вёрстка с классами
- Форматирование: компактное, без лишних пробелов (исторически)
- Язык UI и комментариев: русский

### Ограничения
- Bitrix24 REST API через входящий вебхук — ограниченный набор методов, нет WebSocket, нет подписок
- Файлы загружаются только через `disk.folder.uploadfile` в папку id=3, затем привязываются к задаче
- Наблюдатель — только через отдельный POST после создания, не в момент создания
- Нет авторизации пользователя — вебхук имеет права пользователя, создавшего токен
- Мобильная версия: sidebar скрыт, сетка в одну колонку
